01 큰 그림

코드를 보기 전에, 이 서비스가 무슨 일을 하는지부터.

사용자는 밤에 관심 종목만 등록해두고 잡니다 (예: NVDA, TSLA). 미국 장은 한국 시간 밤에 열리죠. 시스템은 장 마감 후 종목별 뉴스·시세를 모아 AI가 한국어 브리핑을 만들어 저장해둡니다. 아침에 사용자가 앱을 열면 이미 만들어진 브리핑을 그냥 보여주기만 하면 됩니다.

이 구조의 핵심은 "실시간이 아니라 하루 1회 배치 + 캐싱"이라는 점입니다. 그래서 백엔드는 크게 두 부분으로 나뉩니다:

  • ① 실시간 API — 프론트가 부르는 REST 엔드포인트(회원가입, 관심종목, 브리핑 조회). PR #1
  • ② 배치 파이프라인 — 뉴스·시세를 수집하고 LLM으로 브리핑을 생성해 DB에 캐싱. PR #2
왜 이렇게? 브리핑을 종목 단위로 하루 한 번만 만들어 저장하면, 100명이 같은 AAPL을 봐도 LLM 호출은 하루 1회입니다. 비용이 사용자 수와 무관해집니다.

03 Finnhub API 키 발급

뉴스·시세를 가져오려면 무료 키가 필요합니다.

  1. finnhub.io 접속
    우측 상단 Get free API key 또는 Sign up 클릭.
  2. 무료 가입
    이메일·비밀번호만으로 가입됩니다. 신용카드 필요 없음.
  3. Dashboard에서 키 복사
    가입 직후 대시보드에 API Key가 바로 표시됩니다.
  4. .env에 붙여넣기
    backend/.env 파일(없으면 .env.example 복사)에 아래처럼 채웁니다.
  5. 수집 실행
    키를 넣은 뒤 수집 스크립트를 돌리면 price_snapshots·news_articles 테이블이 채워집니다.
# backend/.env
FINNHUB_API_KEY=여기에_복사한_키

# 실행
python -m app.jobs.collect_market_data

05 요청 하나가 처리되는 흐름

구체적인 3가지 예시로 계층이 어떻게 이어지는지 따라가 봅니다.

예시 1 · 회원가입

POST /auth/register  { email, password, nickname }
   │
   routes/auth.py      # 이메일 중복 확인
   crud/user.py        # 비밀번호 bcrypt 해시 후 INSERT
   core/security.py    # JWT 토큰 발급
   ↓
{ access_token, user }   # 프론트는 이 토큰을 저장

예시 2 · 관심종목 등록 (로그인 필요)

POST /watchlist   Authorization: Bearer <token>
   │
   api/deps.py            # get_current_user: 토큰 검증 → 내 User
   crud/stock.py          # 종목이 존재하는지 확인
   crud/watchlist.py      # 중복 아니면 INSERT (UNIQUE 제약)
   ↓
201 Created  { id, ticker, stock }

예시 3 · 오늘의 브리핑 조회 (핵심)

여기가 캐시 + 온디맨드 생성이 함께 도는 부분입니다.

GET /briefings/today   Authorization: Bearer <token>
   │
   crud/watchlist.py          # 내 관심종목 목록
   daily_briefings 조회       # 오늘자 캐시가 있나?
   │
   ├─ 있으면 → 그대로 반환 # LLM 재호출 없음 (싸다)
   └─ 없으면 → services/briefing_pipeline.py
                  # 뉴스 모아 1단계→2단계 → 저장 후 반환
계층의 규칙 routes는 "무엇을" 할지 정하고(입출력·인증), crud/services가 "어떻게" 할지(DB·외부 API)를 담당합니다. 라우터에 DB 쿼리를 직접 쓰지 않고 분리해두면 테스트·재사용이 쉬워집니다.

07 LLM 2단계 파이프라인

이 서비스의 진짜 핵심. 왜 한 번에 안 하고 두 단계로 나눌까?

뉴스 원문
news_articles
→
① 팩트 추출
중립·사실만
→
② 성향 렌더
관점·톤 적용
→
브리핑
JSON 저장
  • 1단계 (extract_facts) — 긴 뉴스에서 사실만 뽑아 FactsExtraction으로. 해석·전망 금지. 무거운 작업이라 한 번만 하고 캐싱.
  • 2단계 (render_briefing) — 그 팩트에 성향(가치투자/모멘텀…)·카테고리를 입혀 BriefingRender로. 짧은 팩트만 재서술하므로 저렴.
2단계인 이유 3가지 ① 비용 — 무거운 원문 읽기는 1회만. ② 환각 방지 — 사실이 1단계에 고정돼 2단계가 없는 수치를 지어낼 수 없음. ③ 맞춤 — 같은 팩트를 성향만 바꿔 여러 버전.

인터페이스로 갈아끼우는 구조

파이프라인은 구체적인 LLM을 모릅니다. BriefingLLMClient라는 추상 인터페이스에만 의존하고, factory.get_llm_client()가 실제 구현을 골라줍니다. LLM API 종류를 아직 안 정했기 때문에 지금은 스텁을 씁니다.

BriefingLLMClient            # 추상 (메서드 2개)
   ├─ StubBriefingLLMClient   # ← 지금 이걸 사용 (더미 데이터)
   └─ ClaudeBriefingLLMClient # 스켈레톤 (나중에 두 메서드만 구현)

# 나중에 API 확정되면 factory.py 에서:
if settings.ANTHROPIC_API_KEY:
    return ClaudeBriefingLLMClient(...)   # 이 줄만 켜면 됨
지금 상태 브리핑 조회는 동작하지만 내용은 "[스텁 브리핑] …" 더미입니다. 구조·저장·API는 실제와 100% 동일해서, Claude 두 메서드만 채우면 그대로 진짜 브리핑이 됩니다.

08 캐싱 전략

비용을 통제하는 장치는 대부분 DB의 UNIQUE 제약입니다.

테이블캐시 키(UNIQUE)효과
daily_briefings(ticker, briefing_date)종목·날짜당 브리핑 1건 → LLM 하루 1회
market_overviewsbriefing_date날짜당 전체 시황 1건
news_articlesurl같은 기사 중복 저장 방지
price_snapshots(ticker, trade_date)종목·날짜당 종가 1건

조회할 때 "오늘자 캐시 있나?" SELECT → 있으면 그대로, 없으면 만들어서 INSERT. DB 제약이 "하루 1건"을 강제하므로 애플리케이션 로직이 단순해집니다.
